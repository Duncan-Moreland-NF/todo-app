1. Overview
The Todo App is a simple, single-user, full-stack task management application. It enables a user to create, view, complete, and delete personal tasks through a clean, responsive web interface backed by a persistent API.
The goal is a focused, reliable, and extensible core product — not a feature-rich platform. Complexity is deliberately minimised in this version.

2. Problem Statement
People need a lightweight, frictionless way to track personal tasks. Existing tools are often over-engineered, require sign-up, or don't work well across devices. This app delivers the essential task management loop — add, view, complete, delete — with zero onboarding and immediate usability.

5.2 Core Features
FR-01 — Create a Todo
The user can add a new todo by typing a task description and submitting it. The todo appears immediately in the list. An empty or whitespace-only title must be rejected.
FR-02 — View All Todos
The user sees all their todos when the app loads. The list is sorted by creation time (newest first or oldest first — to be decided in architecture). The list updates in real time after any action.
FR-03 — Complete a Todo
The user can mark a todo as completed. Completed todos are visually distinct (e.g. strikethrough text, greyed out). Toggling is supported — a completed todo can be marked as incomplete.
FR-04 — Delete a Todo
The user can permanently delete a todo. This action is immediate and does not require a confirmation dialog (simplicity preference for v1).
FR-05 — Empty State
When no todos exist, the app shows a helpful empty state message rather than a blank list.
FR-06 — Error Handling
If an API call fails, the user sees a clear, non-disruptive error message. The app does not crash or show a broken UI.
FR-07 — Loading State
While data is being fetched, a loading indicator is shown so the user knows the app is working.

6. Non-Functional Requirements
NFR-01 — Performance
API responses should return within 200ms under normal local/development conditions. UI interactions should feel instantaneous (optimistic updates are encouraged but not required for v1).
NFR-02 — Responsiveness
The UI must be usable on both desktop (≥ 1024px) and mobile (≥ 375px) screen widths.
NFR-03 — Persistence
Todo data must survive page refreshes and app restarts. Data is stored in a database (not browser storage).
NFR-04 — Simplicity
The codebase should be easy to read and understand. Prefer straightforward patterns over clever abstractions.
NFR-05 — Extensibility
The architecture must not block future addition of: user authentication, multiple users, task categories, or deadlines. These are NOT being built, but the design should leave the door open.
NFR-06 — Deployability
The application must be fully containerisable and run via docker-compose up with no additional manual setup.
NFR-07 — Maintainability
The project must include a README with setup instructions. Code should be linted and consistently formatted.

8. Out of Scope (v1)
The following are explicitly excluded from this version:

❌ User accounts, login, authentication
❌ Multiple users or shared lists
❌ Task categories, tags, or priority levels
❌ Due dates, deadlines, or reminders
❌ Push notifications or email alerts
❌ Drag-and-drop reordering
❌ Search or filtering
❌ Undo/redo
❌ Offline mode
❌ Native mobile apps (iOS/Android)


9. Assumptions and Dependencies

The app will be built as a web application (browser-based)
A single developer (or AI-assisted workflow) is building this
The app does not need to handle high concurrency or scale in v1
Docker Desktop will be available for local deployment testing
Node.js and npm will be the primary runtime environment
